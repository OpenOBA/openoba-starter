# OpenOBA 定位声明

> 用于 README、官网、技术演讲、开源公告等场景
> 核心原则：诚实承认差距，精确指出差异，不夸大不贬低

---

## 一句话定位

OpenOBA 不是更好的 ERP，也不是更强的 Agent——它是一种新的企业数字化系统开发与运营范式：让 AI 住在系统内部，自己开发自己、自己运营自己。

---

## 完整定位描述

### 诚实的差距

在企业数字化业务系统的成熟度上，OpenOBA 与专业系统开发商（SAP、用友、金蝶等）存在显著差距。作为一个历时 120 天、由人机共创完成的项目，它在功能广度、性能压测、生产稳定性、行业覆盖度等方面远不及沉淀了十几年、几千万行代码的成熟商业 ERP。OpenOBA 的 ERP 层代码质量评级为 C+，测试覆盖率约 9%，这些数字诚实地反映了它的早期阶段。

在通用编程 Agent 领域，OpenOBA 与 Cursor、Claude Code 等产品不在同一赛道。后者是面向开发者的 IDE 辅助工具，在代码补全、重构、调试等方面有深厚积累。OpenOBA 的 AI 执行官使用相同的底层大模型（DeepSeek / Qwen），在"写代码的能力"上并无优势。

### 真正的差异

OpenOBA 的价值不在于"更好的 ERP"或"更强的 Agent"，而在于一种**新的范式**。

传统的企业数字化系统开发，无论开源还是商业，都遵循同一套流程：人读文档 → 人搭环境 → 人读源码 → 人写代码 → 人提 PR → 人 Review → 部署。这套流程的瓶颈是人——人的时间、人的能力、人的沟通成本。

OpenOBA 提供了另一条路径：AI 通过 Meta-Mirror 自动理解系统结构，通过 ERDL 规则引擎读写业务规则，通过 file-edit + 编译验证闭环自己改自己的代码，通过 Agent Memory 积累经验不重复犯错，通过 Deliverable 审批流保证安全。人只需要描述"要什么"，AI 自己完成"怎么做"。

这套能力建立在七项发明专利之上，形成从语义地基到业务协同的完整七层架构。它不是一个更快的写代码工具，而是一个**能自己开发自己、自己运营自己的系统**。

### 范式的意义

OpenOBA 的意义不在于它现在的代码质量或功能完整度——这些会随着迭代持续提升。它的意义在于证明了一件事：

> 企业数字化系统的开发和运营，可以不依赖专业的开发团队。一个懂业务的人，加上一个能自开发的 AI 系统，就能完成过去需要一个工程团队才能做的事。

这降低了企业数字化的门槛。不是每个人都请得起 SAP 实施团队，但每个人都可以打开一个对话框，说"我要一个能管商品、管订单、管库存的系统"。

OpenOBA 是这条路径的第一个完整走通的参考实现。它不成熟，但它证明了方向。

## 不同场景的精简版

### GitHub README 开头（3 行）

> OpenOBA Starter 是一个自指的 ERP 系统：AI 执行官住在 ERA-Chat 里，能读自己的代码、改自己的代码、运营自己的业务。这不是一个更快的写代码工具，而是一种新的企业数字化范式——让系统自己开发自己。代码完全由 AI 生成，由一个不懂开发的人与 AI 共创而成。

### 技术演讲开场（30 秒）

> OpenOBA 不是一个要和 SAP 比功能的 ERP，也不是要和 Cursor 比写代码的 Agent。它回答的是另一个问题：如果 AI 能自己读懂系统、自己改代码、自己跑验证、自己运营业务，那企业数字化系统的开发和运营还需要专业的开发团队吗？OpenOBA 用 7 项专利和 120 天的人机共创，给出了一个肯定的答案：不需要。

### 开源公告（1 段）

> 我们开源 OpenOBA Starter，不是为了提供又一个 ERP 系统，而是为了分享一种新的企业数字化范式。这个项目的代码没有一行是人类写的——它由一个完全不懂开发的项目发起人，与 DeepSeek V4 Pro / Qwen 3.6 历时 120 天、上百轮对话共创而成。它不成熟，功能远不及成熟商业 ERP，代码质量还有很大提升空间。但它证明了一件事：让 AI 住在系统内部、自己开发自己、自己运营自己，这条路是走得通的。七项发明专利支撑的七层架构，是这条路的地基。我们把它开源出来，希望更多人和更多行业一起走这条路。

## 不该怎么说（反模式）

在描述 OpenOBA 时，避免以下表述：

| 不要说 | 应该说 | 原因 |
|--------|--------|------|
| "比 SAP 更好的 ERP" | "一种新的企业数字化范式" | 功能差距客观存在，不比功能 |
| "比 Cursor 更强的 Agent" | "不同品类的系统——自指 Agent" | 不在同一赛道 |
| "AI 自主开发" | "AI 自开发（人在审批环上）" | 人仍需 Review Deliverable |
| "零代码开发" | "自然语言开发" | 仍有技术门槛（需理解业务） |
| "企业数字化基座" | "企业数字化开发与运营新框架" | "基座"暗示底层基础设施，目前不到这个程度 |
| "革命性" / "颠覆性" | "新的范式" / "新的路径" | 让事实说话，不用营销词 |

## 技术实现的支撑

这段定位不是营销话术，有技术实现支撑：

| 定位主张 | 技术证据 |
|---------|---------|
| "AI 能读自己的代码" | Meta-Mirror 6 个 scanner 自动扫描 Entity/API/DTO/规则/约定 |
| "AI 能改自己的代码" | file-edit skill + allowPatterns 权限白名单 |
| "改完自动验证" | P01 编译验证闭环：file-edit 自动调 tsc，错误自动修正重试 |
| "AI 能运营业务" | 18 个 skill + 六引擎协同（P07） |
| "AI 不重复犯错" | P06 错误→记忆转化链 + 按作用域精准注入 |
| "安全可控" | 五道防线：SOUL 铁律 + Skill 白名单 + 编译验证 + 三重数据库校验 + Deliverable 审批 |
| "这是新范式" | 7 项发明专利，专利摸底报告确认新颖性 |
