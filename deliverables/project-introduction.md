# OpenOBA 项目简介

> 版本：v1.0 · 2026-06-24
> 用途：项目对外介绍的标准文档，适用于官网、投资材料、合作洽谈、媒体介绍等场景

---

## 一句话介绍

OpenOBA 是一种新的企业数字化系统开发与运营范式：让 AI 住在系统内部，通过自然语言对话自己开发自己、自己运营自己。

---

## 什么是 OpenOBA

OpenOBA（Open Business Automation）是一个**自指（self-referential）的企业数字化系统**。和传统 ERP 不同，它的核心不是一套表单和报表，而是一个住在系统内部的 AI 执行官——这个 AI 通过 Meta-Mirror 引擎读懂系统自身的代码结构，通过 file-edit 改写系统自身的代码，通过编译验证闭环自动检查改动，通过 Agent Memory 积累经验不重复犯错。

用户不需要写代码，也不需要理解系统架构。只需要在 ERA-Chat 对话界面里用自然语言描述需求——"在客户表加一个生日字段"、"查一下本月退货率最高的 SKU"、"给售后模块加一个 7 天无理由自动审批规则"——AI 执行官会自己读代码、改代码、跑验证、运营业务。

这套能力建立在 **7 项发明专利**之上，形成从语义地基到业务协同的完整七层架构。

## 核心特征

### 1. 自指架构：AI 读自己的代码、改自己的代码

传统 AI 助手不知道你项目里有什么——你得手动粘贴代码。OpenOBA 的 Meta-Mirror 引擎在启动时自动扫描整个系统的 Entity、API、DTO、业务规则、命名约定，生成知识库注入到每个 AI Agent 的上下文里。AI 能说出"我可以通过 InventoryService.lowStockQuery() 查低库存"，因为它扫到了这个 API。

改代码时，AI 用 file-edit 修改源文件，改完**自动调用 TypeScript 编译器**做类型检查。编译失败时 AI 根据错误信息自我修正后重试，循环直至通过——整个过程无需人工干预。

### 2. 七项发明专利支撑的七层架构

| 层 | 引擎 | 专利 | 解决的问题 |
|----|------|------|-----------|
| L1 | ERDL 规则与数据操作 | P01+P04 | AI 编造字段名 / SQL 注入 → 确定性四步翻译链 |
| L2 | Action Guard 协议转换 | P02 | LLM 输出格式飘忽 → 四格式自适应解析 |
| L3 | ReAct 单工具决策 | P03 | 并行工具调用的浪费 → 每轮单工具基于真实结果再决策 |
| L4 | SOUL 人格系统 | — | AI 行为不可控 → 身份+能力边界+铁律三层约束 |
| L5 | Meta-Mirror 自省 | — | AI 不了解系统 → 自动扫描注入 |
| L6 | Agent Memory | P06 | AI 每次从零开始 → 错误自动转为持久记忆 |
| L7 | 六引擎协同运营 | P07 | 模块割裂 → 单 Agent 全业务闭环 |

### 3. 人机共创：代码完全由 AI 生成

这个项目没有一行人类代码。它由一个完全不懂开发的项目发起人，与 DeepSeek V4 Pro / Qwen 3.6 历时近 120 天、上百轮对话共创而成。发起人负责"业务该是什么样"的判断，AI 负责"怎么实现"的工程能力。7 项专利的技术方案均由 AI 起草，专利署名栏写着"起草人：OpenOBA（唐浩然 · AI 执行官）"。

### 4. 完整的眼镜行业 ERP

OpenOBA 不仅仅是一个 AI 引擎，它还包含一个完整的眼镜行业 ERP 系统——商品管理（SPU/SKU/套装/条码）、客户管理、订单管理、库存管理、售后管理、评价系统、短信系统、美学搭配规则、颜色设计、镜架结构标准。128 张数据库表，21 个 NestJS 业务模块，传统表单操作全部保留。

## 与传统方案的差异

### 与传统开源 ERP 的差异

| 维度 | 传统开源 ERP | OpenOBA |
|------|------------|---------|
| 二开入口 | README + 源码 | ERA-Chat 对话框 |
| 谁读代码 | 开发者 | AI（Meta-Mirror 自动扫描注入） |
| 谁改代码 | 开发者写 PR | AI 用 file-edit 改，自动编译验证闭环 |
| 规则变更 | 改代码重新部署 | 写 ERDL YAML 热加载 |
| 交付单位 | Git Commit / PR | Deliverable（含 changelog + 文件清单 + 版本） |

### 与通用编程 Agent（Cursor / Claude Code）的差异

OpenOBA 与 Cursor、Claude Code 不在同一赛道。后者是面向开发者的 IDE 辅助工具；OpenOBA 是住在企业系统内部的 AI 执行官。纯比"写代码能力"，OpenOBA 用的也是 DeepSeek/Qwen，并无优势。差异在于：Cursor 是"外部工具帮你改代码"，OpenOBA 是"系统内部的 AI 自己改自己"——AI 通过 Meta-Mirror 自动理解系统结构，通过编译验证闭环自动验证，通过 Agent Memory 积累经验。

### 诚实的差距

在企业数字化业务系统的成熟度上，OpenOBA 与 SAP、用友、金蝶等存在显著差距。作为一个历时 120 天、由人机共创完成的项目，它在功能广度、性能压测、生产稳定性、行业覆盖度等方面远不及沉淀了十几年的成熟商业 ERP。OpenOBA 的 ERP 层代码质量评级为 C+，测试覆盖率约 9%——这些数字诚实地反映了它的早期阶段。

OpenOBA 的价值不在于"更好的 ERP"或"更强的 Agent"，而在于一种**新的范式**。

## 安全机制

让 AI 能改自己的代码，安全是首要问题。OpenOBA 用五道防线约束：

| 防线 | 机制 | 作用 |
|------|------|------|
| 1. SOUL 铁律 | securityClearance + canEditCode + 铁律文本 | 低权限 Agent 看不到危险 skill |
| 2. Skill 权限白名单 | file-edit 的 allowPatterns 限定 src/skills/erdl/packages | 碰不到 .env / node_modules |
| 3. 编译验证闭环 | file-edit 改完自动跑 tsc，错误自动修正 | 不会带着类型错误交付 |
| 4. 三重数据库校验 | 白名单 + 禁止表 + 只读保护 | 级联校验任一失败立即终止 |
| 5. Deliverable 审批流 | AI 改完打包，必须人工 Review | AI 不能直接上线 |

每次 tool 调用写 cognitive_log 审计日志，全程可追溯。

## 项目背景

OpenOBA 是一个完全的人机共创产品。项目发起人没有开发经验，最初只是想用 AI 为自己的眼镜电商手搓一个 ERP。经过与 AI 的数百轮会话，决定投入精力开发 OpenOBA。历时近 120 天、上百轮修改和重构，从 v0.1.0（2026-04-05）到 v1.4.0-alpha9（2026-06-23），产品代码完全由 DeepSeek V4 Pro 完成，部分 ERP 代码由 Qwen 3.6 实现。

## 适用场景

OpenOBA 当前以眼镜行业为参考实现，但其七层架构的 L1-L4（ERDL / Action Guard / ReAct / SOUL）是行业无关的框架。通过编写不同的 ERDL 定义文件，可适用于不同行业领域。

**适合谁**：
- 想体验"AI 自开发自运营"新范式的技术团队
- 需要快速搭建行业 ERP 的中小企业
- 研究 AI Agent 与企业系统集成的技术团队
- 对人机共创模式感兴趣的研究者

**不适合谁（目前）**：
- 需要生产级稳定性的大型企业（建议等 v2.0+）
- 需要成熟财务模块的企业（当前无财务模块）
- 对代码质量有严格要求的团队（当前 C+ 评级，9% 测试覆盖率）
