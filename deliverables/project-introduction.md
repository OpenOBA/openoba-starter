## 什么是 OpenOBA

OpenOBA（Open Business Automation）是一个**AI 数字执行官系统**。

它和传统 ERP 最大的不同：传统 ERP 是一个工具——你登录系统，手动点菜单、填表单、查报表。OpenOBA 是一个执行官——你跟它对话，它自己操作数据库、处理表单、执行业务、生成报表、甚至改写自己的代码。

### 运营能力：AI 能直接操作你的业务

OpenOBA 的 AI 不是"给你建议"——它真的能动手。

- **查数据**：说"查一下本月退货率最高的 5 个 SKU"，Agent 自己写查询、走数据库、返回结果表格
- **操作表单**：说"把客户 Henry 的会员等级升为 VIP"，Agent 自己找到客户记录、修改字段、记录操作日志
- **处理订单**：说"今天上午付过款还没发货的订单列出来，全部发圆通"，Agent 逐条处理
- **导入导出**：说"把本月销售数据导出为 Excel"，Agent 生成文件
- **生成报表**：说"给我一份本季度会员消费排行榜"，Agent 自己统计、排序、排版
- **抓取分析**：说"抓一下这个竞品页面，分析他们的定价策略"，Agent 抓取、解析、对比
- **自动运营**：设定规则后，Agent 自动巡检库存不足的 SKU、自动下架过期活动、自动处理 7 天无理由退货

这些不是"未来的计划"——**它已经在运行**。18 个 Skill 覆盖了日常运营的全链路。

### 开发能力：AI 能改写自己的代码

更独特的是，OpenOBA 的 AI 能自己改自己的代码。

- **添加字段**：说"在客户表加一个生日字段"，Agent 自己读 Entity、改代码、跑编译验证、打包交付——**全程无人干预**
- **创建模块**：说"给我加一个供应商管理模块，要有增删改查"，Agent 从 Entity 到 Service 到 Controller 到前端页面，一行行自己写
- **调整定价规则**：说"满 200 减 30 的促销改成满 300 减 50"，Agent 自己找到规则定义、修改、验证不破坏类型系统
- **修复 Bug**：遇到 BUG，Agent 自己读错误日志、定位根因、修改代码、编译验证、交付修复包

**AI 不是在"帮你写代码"——AI 是系统的一部分，它自己写自己。**

## 核心特征

### 1. 全链路运营执行

| 场景 | 你的话 | Agent 做的事 |
|------|--------|-------------|
| 数据查询 | "本月退货率最高的 SKU" | 生成 SQL → 查询 MySQL → 返回表格 |
| 表单操作 | "把 Henry 升为 VIP" | 查找客户记录 → 更新字段 → 写操作日志 |
| 批量处理 | "所有待发货订单发圆通" | 分页查询 → 逐条更新物流信息 |
| 导入导出 | "导出月销售报表" | 查询数据 → 生成 Excel → 提供下载 |
| 自动监控 | "库存低于 10 的 SKU 每天通知我" | 定时巡检 → 生成告警列表 |
| 网页抓取 | "抓竞品页面分析定价" | HTTP 请求 → HTML 解析 → 对比分析 |

### 2. 自指开发架构

```
用户："在客户表加一个生日字段"
  ↓
Agent (Thought): 我需要先看 Customer Entity 的结构
Agent (Tool): Meta-Mirror → 读取 Customer Entity → 有 id/name/phone/email，无 birthday
Agent (Thought): 用 file-edit 添加 birthday 字段
Agent (Tool): file_edit → 改代码 → 自动跑 tsc → ✅ 0 errors
Agent (Tool): git_diff → + @Column({ nullable: true }) birthday?: Date
Agent: 已添加，类型检查通过。Diff 已打包为 Deliverable，请 Review 后发布。
```

**这不是 AI 辅助开发——是 AI 自己开发自己。**

### 3. 七项发明专利支撑

| 层 | 引擎 | 解决的问题 |
|----|------|-----------|
| L1 | ERDL 规则引擎（P01+P04） | AI 编造字段名 / SQL 注入 → 确定性四步翻译链 + 编译验证闭环 |
| L2 | Action Guard（P02） | LLM 输出格式飘忽 → 四格式自适应解析 |
| L3 | ReAct 单工具（P03） | 并行工具调用浪费 → 每轮单工具，基于真实结果决策 |
| L4 | SOUL 人格系统 | AI 行为不可控 → 身份 + 权限 + 铁律三层约束 |
| L5 | Meta-Mirror 自省 | AI 不了解系统 → 自动扫描 Entity/API/DTO/规则注入 |
| L6 | Agent Memory（P06） | AI 每次从零开始 → 错误自动转为持久记忆 |
| L7 | 六引擎协同（P07） | 模块割裂 → 单 Agent 全业务闭环 |

### 4. 人机共创：0 行人类代码

120 天，332 次提交，7 项发明专利，0 行人类代码。完全由项目发起人（不懂开发）与 DeepSeek V4 Pro / Qwen 3.6 对话共创。

### 5. 完整的眼镜行业 ERP

128 张数据库表，21 个 NestJS 业务模块——商品（SPU/SKU/套装/条码）、客户、订单、库存、售后、评价、会员、营销、颜色设计、镜架结构标准。传统表单操作全部保留，AI 数字执行官与传统 UI 无缝衔接。

## 安全机制

让 AI 能直接操作数据库、改代码，安全是生死线。OpenOBA 用五道防线：

| 防线 | 机制 | 作用 |
|------|------|------|
| 1. SOUL 铁律 | 权限白名单 + 行为约束 | 低权限 Agent 看不到危险 Skill |
| 2. 编译验证闭环 | 改完自动跑 tsc，失败自动修正 | 不会带着类型错误上线 |
| 3. 三重数据库校验 | 白名单 + 禁止表 + 只读保护 | 级联校验，任一失败立即终止 |
| 4. 认知审计日志 | 每次操作写 cognitive_log | 全链路可追溯 |
| 5. Deliverable 审批 | AI 改完打包，必须人工 Review | AI 不能直接上线 |
