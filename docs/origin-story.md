# OpenOBA 诞生记：当 AI 学会改自己的代码

> 一个完全不懂代码的人，用 AI 造了一个能自己改自己代码的 ERP。
> 这不是营销话术——这是 120 天，308 次提交，7 项发明专利的真实记录。

---

## 序

我叫 Henry。我不会写代码。一行都不会。不知道 TypeScript 是什么，不知道 NestJS 是干什么的，不知道 Vue 3 和 Vue 2 有什么区别。

但我懂眼镜行业。我知道一副镜架怎么从供应商到仓库，怎么从仓库到顾客手里。我知道退换货怎么处理，库存周转怎么看，会员等级怎么算。

2026 年 4 月，我问了 AI 一个问题。

这个问题的答案，变成了一个开着 128 张数据库表、跑着 21 个 NestJS 模块、有 7 项发明专利、能自己改自己代码的 ERP 系统。

---

## 第一章：初衷 —— 我只是想要一个 ERP

### 2026-04-05 · 项目初始化

最初的想法很简单：用 AI 帮我写一个自用的眼镜电商 ERP。

市面上不是没有 ERP。SAP、用友、金蝶，哪一个都比我手写的强大一万倍。但它们有一个共同的问题：我需要会写代码才能定制它。

而我不会。

于是我打开对话框，对 AI 说：

> "帮我搞一个能管理商品、管理订单、管理库存的系统。"

AI 开始写代码。第一版很简单——几张表，几个接口，一个能看能改的前端页面。

**第一次提交：4810798，V1.3.0 基线。**

那时候还不知道，这一步迈出去，就不只是"写一个 ERP"这么简单了。

### 2026-04-05 ~ 2026-05-31 · 基础搭建

第一个月是"能用"阶段。把眼镜行业的核心业务流程——商品（SPU/SKU/套装/条码）、客户、订单、库存、售后——一个模块一个模块搭起来。

134 张数据库表。每张表的结构都是我在对话框里描述的："客户表应该有姓名、电话、邮箱、地址、等级……"AI 把它翻译成 TypeORM Entity，生成 NestJS 模块，搭出 Vue 3 前端页面。

磕磕绊绊。建表建错了，字段类型不对，关系搞反了——每次出错，我都截图告诉 AI，AI 修，我再试。

到 5 月底，一个初具规模的 ERP 立起来了。

但也只是"一个 ERP"。它和你在 GitHub 上能找到的几十个开源 ERP 没有本质区别——表单 UI，CRUD 接口，查询列表。

真正的转折，从下一个月开始。

## 第二章：转型 —— 让 AI 自己运营这个系统

### 2026-06-01 ~ 2026-06-09 · Agent 实验期

ERP 做好了，但有个问题：我得自己去用它。

我要每天登录系统，查哪些订单没发货，手动处理售后，手动更新库存。一个不会写代码的人，自己运营一个自己造的 ERP——这听起来绕口，但就是我的处境。

我想到：能不能让 AI 帮我运营它？

于是我开始了一个实验：给 AI 装上 Skill。

Skill 是一种工具注册机制。你可以注册一个"查所有待发货订单"的 Skill，Agent 就能在任何对话中调用它。

6 月初，我注册了第一批 Skill：查商品、查订单、查客户、改库存。

然后我对 AI 说："查一下今天有多少待发货的订单，列出来。"

AI 自己调了 `queryOrders` Skill，拿到了数据，用自然语言回复了我。

**这是第一次。AI 不是在帮我写代码，而是在帮我干业务。**

那一刻，我意识到这件事和"AI 写代码"是两码事。

### 产品定位的第一次调整

原计划：做一个眼镜行业 ERP。

新方向：做一个能让 AI Agent 自主运营业务的系统。

但这时候名字还是乱的——秒镜、WorkBuddy、ERA-Chat、唐浩然……各种名字混在一起。产品形态也不清晰。

## 第三章：定型 —— 五层创新架构的浮现

### 2026-06-09 ~ 2026-06-15 · 架构爆发期

让 Agent 运营 ERP，听起来简单，做起来全是坑。

#### 坑 1：AI 老编造字段名

我对 AI 说"查一下客户的订单"，AI 调 `queryOrders` 时自己编了个 `where: { userId: 123 }`——但数据库里根本没有 `userId` 这个字段。

解决方案？不能让 AI 靠记忆记住 134 张表的结构。得让 AI 自己"看到"系统的结构。

这就是 **Meta-Mirror**（元镜引擎）的起点：启动时自动扫描系统所有的 Entity、API、DTO、业务规则，生成知识库，自动注入到每个 AI Agent 的上下文里。AI 能说出"我可以通过 `InventoryService.lowStockQuery()` 查低库存"，因为它扫到了这个 API。

#### 坑 2：AI 输出格式飘忽不定，系统老崩

有时候 LLM 返回的是 Function Call JSON，有时候是纯文本，有时候是格式乱的 JSON，有时候直接输出注释。

你不能指望 LLM 每次都按你期望的格式输出。得有一个协议层来"接住"它无论怎么输出，都翻译成系统能理解的动作。

这就是 **Action Guard**（行动卫兵）的起点：四格式自适应解析（FC/JSON/XML/文本）+ 三级校验 + 自动清洁 + 失败回退。

#### 坑 3：AI 一轮做太多事，容易出错

早期 Agent 一个回合就会返回多个工具调用——"同时查商品、查订单、更新库存"。但三件事有一个失败了，整个回合都崩了。

改成每轮只做一个决策，基于真实结果再决定下一步。

这就是 **ReAct 单工具决策**的起点。

#### 坑 4：AI 改完代码不知道对不对

当 Agent 开始改自己的代码时，改完了它不知道有没有破坏类型系统。得让它改完自动跑 TypeScript 编译器——编译失败就自己修正重试，循环到通过为止。

这就是 **ERDL 编译验证闭环**的起点。

#### 坑 5：AI 每次从零开始，重复犯错

同一个 Agent，处理类似的订单问题，今天碰到的坑明天继续掉。得让错误转化为记忆——下次处理同类任务时，AI 自动想起"上次在这里踩过坑，别这么干"。

这就是 **Agent Memory**（认知记忆系统）的起点。

## 第四章：定名 —— OpenOBA 的诞生

### 2026-06-15 ~ 2026-06-19 · 品牌统一期

之前的产品名称混乱：

- "秒镜"——最初的商业主体名称，但太垂直
- "WorkBuddy"——内部工程工具名，不能代表产品
- "ERA-Chat"——对话工作台，是组件但不是品牌
- "唐浩然"——我的 AI 执行官身份

Henry 做了关键决策：**统一为 OpenOBA。**

```
Open  = 开放
O     = Operations（运营）
B     = Business（开发）
A     = Agent（智能体）
```

**OpenOBA = AI 原生自主执行体。**

这个命名定义了四件事：

1. **Open** —— 开源，开放
2. **运营** —— AI 不只写代码，更要能运营业务
3. **开发** —— AI 能自己开发自己
4. **智能体** —— 不是工具，是执行官

## 第六章：新范式 —— OpenOBA 是什么

OpenOBA 不是一个更好的 ERP，也不是一个更强的 Agent。

它是一种新的企业数字化系统开发与运营范式。

传统范式：
> 人读文档 → 人搭环境 → 人读源码 → 人写代码 → 人提 PR → 人 Review → 部署

OpenOBA 范式：
> 人描述需求 → AI 读系统结构 → AI 改代码 → AI 自动验证 → AI 打包交付 → 人 Review 后发布

**人不写代码。人只做判断。**

这降低了企业数字化的门槛。不是每个人都请得起 SAP 实施团队，但每个人都可以打开一个对话框，说"我要一个能管商品、管订单、管库存的系统"。

### 诚实的部分

OpenOBA 现在的代码质量是 C+，测试覆盖率是 9%。和 SAP 比功能，差得远。和 Cursor 比写代码，也没优势——大家都用 DeepSeek / Qwen。

**但它证明了方向。**

120 天，308 次提交，7 项发明专利，0 行人类代码。

这条路，走得通。
